# NCP HTTP API

The `ncp serve` runtime speaks JSON-RPC 2.0 over plain HTTP POST, with an SSE
discovery stream. Any language with an HTTP client can drive it тАФ MCP hosts
use this transport automatically, but the contract below is public and stable
for programmatic clients (TypeScript orchestrators, shell scripts, CI jobs).

All examples assume:

```bash
ncp serve --host 127.0.0.1 --port 4242 --cwd /path/to/project
BASE=http://127.0.0.1:4242
```

## Endpoints

| Method | Path       | Purpose                                        |
|--------|------------|------------------------------------------------|
| GET    | `/healthz` | Liveness probe (no auth required)              |
| GET    | `/sse`     | SSE discovery stream (emits the RPC endpoint)  |
| POST   | `/mcp`     | JSON-RPC 2.0 requests (also accepts `/message`)|

## Authentication

No token is required on loopback by default. When `[server].auth_token` is set
in `.ncp/config.toml` (generated by `ncp init`), via `NCP_AUTH_TOKEN`, or via
`ncp serve --auth-token`, every `/mcp` and `/sse` request must carry:

```bash
AUTH="Authorization: Bearer $TOKEN"
```

Unauthenticated requests receive `401 {"error": "unauthorized"}`. Request
bodies over 10 MB receive `413`. CORS headers are only emitted for origins
passed via `--cors-origin` (none by default тАФ browsers cannot call the API).

## Health check

```bash
curl -s $BASE/healthz
# {"ok": true, "transport": "http_sse", "rpc_path": "/mcp", "sse_path": "/sse"}
```

## List tools

```bash
curl -s $BASE/mcp -H "$AUTH" -H 'Content-Type: application/json' -d '{
  "jsonrpc": "2.0", "id": 1, "method": "tools/list"
}'
```

Returns the five tools: `ncp_get_context`, `ncp_write_memory`,
`ncp_emit_whisper`, `ncp_post_turn`, `ncp_fetch`, each with a JSON Schema
under `inputSchema`.

## Tool calls

All tool calls share one shape; results arrive as MCP content with a single
JSON-encoded text item:

```json
{"jsonrpc": "2.0", "id": N, "method": "tools/call",
 "params": {"name": "<tool>", "arguments": { ... }}}
```

### ncp_get_context тАФ assemble the bounded turn context

```bash
curl -s $BASE/mcp -H "$AUTH" -H 'Content-Type: application/json' -d '{
  "jsonrpc": "2.0", "id": 2, "method": "tools/call",
  "params": {"name": "ncp_get_context", "arguments": {
    "agent_id": "executor", "role": "build",
    "task": "fix_payment_bug", "slot": "payment", "intent": "advance",
    "pipeline_id": "pipe_demo",
    "max_tokens": 840, "ctx_used": 0.4
  }}}'
```

The decoded result contains:

- `context` тАФ the assembled pidgin block to inject into the model prompt
- `session_id` тАФ pass to `ncp_fetch` to share the fetch budget
- `pending_whisper_ids` тАФ whisper ids consumed this turn; pass them to
  `ncp_post_turn` to acknowledge (at-least-once delivery: unacked whispers
  are redelivered next turn)
- `telemetry` тАФ eviction counts, `fetch_budget_remaining`, and a `fetch_hint`
  when relevant context was evicted under the token budget

Conscious state (`recent`, `tried`, `failed`, `drift_score`, тАж) is hydrated
server-side from the latest `ncp_post_turn` snapshot for this
`(pipeline_id, agent_id)`; explicit arguments override hydrated values.

### ncp_post_turn тАФ record the turn and acknowledge whispers

```bash
curl -s $BASE/mcp -H "$AUTH" -H 'Content-Type: application/json' -d '{
  "jsonrpc": "2.0", "id": 3, "method": "tools/call",
  "params": {"name": "ncp_post_turn", "arguments": {
    "agent_id": "executor", "role": "build",
    "task": "fix_payment_bug", "slot": "payment", "intent": "advance",
    "pipeline_id": "pipe_demo",
    "result_summary": "null_guard_added_at_line_142",
    "result_full": "Applied null guard at PaymentProcessor.java:142; tests pass.",
    "ack_whisper_ids": []
  }}}'
```

Returns `{"posted": true, "turn_id": "..."}`. This is what makes recent-ref
continuity, `ncp cost`, and `ncp status` work.

### ncp_write_memory тАФ persist a durable chunk

```bash
curl -s $BASE/mcp -H "$AUTH" -H 'Content-Type: application/json' -d '{
  "jsonrpc": "2.0", "id": 4, "method": "tools/call",
  "params": {"name": "ncp_write_memory", "arguments": {
    "content": "constraint: retryCount may be null for ACH trial customers",
    "layer": "semantic", "src": "tool_result",
    "written_by": "executor", "pipeline_id": "pipe_demo"
  }}}'
```

`base_trust` defaults by `src` (`user_verified` 0.95, `tool_result` 0.80,
`synthesis` 0.70, `agent_inferred` 0.60) and may be set explicitly.

### ncp_emit_whisper тАФ signal another agent

```bash
curl -s $BASE/mcp -H "$AUTH" -H 'Content-Type: application/json' -d '{
  "jsonrpc": "2.0", "id": 5, "method": "tools/call",
  "params": {"name": "ncp_emit_whisper", "arguments": {
    "from": "executor", "target": "reviewer", "type": "share",
    "payload": "fix applied to PaymentProcessor.java, please review",
    "confidence": 0.9, "pipeline_id": "pipe_demo", "ttl_seconds": 1800
  }}}'
```

`target: "*"` broadcasts to every agent in the pipeline (each receives it
once). Plain-text payloads for `share`/`request`/`dissent` are wrapped into
the required JSON shape server-side.

### ncp_fetch тАФ bounded mid-turn retrieval

```bash
curl -s $BASE/mcp -H "$AUTH" -H 'Content-Type: application/json' -d '{
  "jsonrpc": "2.0", "id": 6, "method": "tools/call",
  "params": {"name": "ncp_fetch", "arguments": {
    "query": "retryCount null guard constraint",
    "agent_id": "executor", "pipeline_id": "pipe_demo", "k": 2
  }}}'
```

Limited to 3 calls per `ncp_get_context` session.

## Streaming

Pass `"stream": true` to `ncp_get_context`. Over HTTP the response is NDJSON:
one `{"type": "ncp_chunk", "section": ..., "index": ..., "text": ...}` line
per section (`conscious`, `subconscious`, `whispers`, `budget_header`),
terminated by the standard JSON-RPC result line. All sections derive from a
single assembly pass, so the concatenated stream matches the non-streaming
`context` exactly.

## Content negotiation (SSE responses)

`POST /mcp` honors the `Accept` header so the endpoint behaves as a stateless
[Streamable HTTP](https://modelcontextprotocol.io) MCP server:

- `Accept: application/json` (or any value that includes it) тАФ the JSON-RPC
  result is returned in the response body. This is the default and matches
  what the MCP SDK sends, so MCP hosts work without extra configuration.
- `Accept: text/event-stream` (without `application/json`) тАФ the response is
  an SSE stream. A normal call yields a single `event: message` carrying the
  JSON-RPC result; a `"stream": true` call yields one `event: ncp_chunk` per
  section followed by the `event: message` result.

There is no `Mcp-Session-Id` header: each `POST /mcp` is self-contained. The
`GET /sse` endpoint is discovery-only тАФ it advertises the RPC path and emits
keepalives, but does not carry JSON-RPC responses.

## Errors

| Condition              | Response                                          |
|------------------------|---------------------------------------------------|
| Unknown tool           | JSON-RPC error `-32601`                           |
| Tool/validation error  | JSON-RPC error `-32603` with the message          |
| Missing/wrong token    | HTTP 401                                          |
| Body too large (>10MB) | HTTP 413                                          |
| Malformed JSON         | HTTP 400                                          |
